      // CRITICAL: log to stderr, NOT stdout.

          .unwrap_or_else(|| PathBuf::from("."));                           
 
                                                                            
 
      let server = GitForensicsServer::new(repo_path);                      
 
                                                                            
 
      // .serve(stdio()) starts the JSON-RPC loop over stdin/stdout         
 
      let service = server.serve(stdio()).await?;                           
 
                                                                            
 
      // Block until the client disconnects                                 
 
      service.waiting().await?;                                             
 
                                                                            
 
      Ok(())

  Key things to understand in this code:                                    
 
                                                                            
 
  1. The macro chain: #[tool_router] on the impl block scans for #[tool]    
 
  methods and generates a Self::tool_router() function. #[tool_handler] on  
 
  the ServerHandler impl wires that router into handle_request.             
 
                                                                            
 
  2. Parameters<T>: This is an extractor. When Claude calls your tool with  
 
  JSON args {"path": "src/main.rs"}, rmcp deserializes it into your         
 
  BlameRequest struct automatically. The JsonSchema derive generates the    
 
  schema that Claude sees.                                                  
 
                                                                            
 
  3. spawn_blocking: git2::Repository is !Send (because libgit2 uses        
 
  thread-local state). You can't hold it across .await points. The pattern  
 
  is: clone what you need, move it into spawn_blocking, do all git work     
 
  there, return a result.                                                   
 
                                                                            
 
  4. stderr for logging: The stdio transport uses stdout for JSON-RPC       
 
  messages. If you println! anything, it corrupts the protocol. Always log 
to
   stderr.                                                                  
 
                                                                            
 
  5. CallToolResult::success: Wraps your output in the MCP content format.  
 
  Claude sees this as the tool's response.

     Build a Git Forensics MCP Server in Rust — Teaching Plan

     Philosophy

     - You write the code, I explain the concepts
     - Each step builds on the last — no jumping ahead
     - We pause after each step so you understand before moving on

     What we're building

     A Git Forensics MCP server with 3 tools:
     1. blame — show who last touched each line of a file
     2. file_history — show commit log for a specific file
     3. hotspots — find files that change most often (churn analysis)

     Lesson Flow

     Step 1: Dependencies & project setup

     - Edit Cargo.toml together — explain each crate and why it's needed
     - rmcp (the MCP SDK), git2 (libgit2 bindings), tokio, serde, schemars
     - Teach: what features are and why we pick specific ones

     Step 2: The server struct + empty main

     - Define GitForensicsServer with a repo_path and a tool_router
     - Write main() with stderr logging + stdio transport
     - Teach: why stderr (stdout = JSON-RPC), what serve(stdio()) does
     - Checkpoint: cargo build should compile

     Step 3: ServerHandler trait — get_info()

     - Implement ServerHandler with #[tool_handler]
     - Teach: what the MCP handshake looks like, what get_info tells the 
     client
     - Checkpoint: still compiles

     Step 4: First tool — blame

     - Define BlameRequest struct with serde + schemars derives
     - Write the #[tool] function using git2::Repository::blame_file

     - Async/await with tokio

  What each crate does:

  | Crate                        | Purpose                                                                                                                                                       |
  |------------------------------|---------------------------------------------------------------------------------------------------------------------------------------------------------------|
  | rmcp                         | The MCP SDK. "server" gives us the server traits/macros. "transport-io" gives us stdio() — the transport that reads JSON-RPC from stdin and writes to stdout. |
  | tokio                        | Async runtime. MCP servers are async because they handle concurrent requests. "full" enables everything (timers, IO, spawn_blocking).                         |
  | git2                         | Rust bindings to libgit2. This is how we read git repos without shelling out to the git CLI.                                                                  |
  | serde                        | Serialization framework. "derive" lets us write #[derive(Deserialize)] on structs so JSON automatically maps to Rust types.                                   |
  | schemars                     | Generates JSON Schema from Rust types. MCP requires tools to declare their parameter schemas — this auto-generates them from your structs.                    |
  | tracing / tracing-subscriber | Structured logging. We log to stderr (critical — stdout is the JSON-RPC transport, one stray println! corrupts the protocol).                                 |
  | anyhow                       | Ergonomic error handling. Lets main() return Result with any error type.                                                                                      |

  Also change the edition — 2024 is bleeding edge, 2021 is what all these crates target:

  edition = "2021"

  Go ahead and make those edits, then run cargo check (not build — it's faster, just type-checks). It'll download all the crates and make sure they resolve. Tell me what happens.
